## ARCHITECTURE DECISIONS

Before reading the tree, understand the four core rules that govern where every file lives:

**Rule 1 — Route Groups for Admin**
`app/admin/(protected)/` is a route group. The `(protected)` folder name does NOT appear in the URL. It exists only to apply the sidebar layout to all admin pages inside it. The login page lives at `app/admin/login/` — outside this group — so it gets no sidebar.

**Rule 2 — Feature-Based Colocation**
Each domain feature (leads, portfolio, faq, settings, seo) owns its own folder inside `src/features/`. That folder contains the Server Actions, database queries, TypeScript types, Zod validation schemas, and constants for that feature. Everything for one feature stays together.

**Rule 3 — `lib/` is Infrastructure Only**
`src/lib/` holds client setup for third-party tools (Prisma, Supabase, Auth.js, Resend, Turnstile). No business logic lives here — only the plumbing.

**Rule 4 — `config/` is Pure Data**
`src/config/` holds static data objects: the city list, navigation links, service definitions. No functions, no async calls, no database access. Just plain data your components import.

## FOLDER PURPOSE QUICK REFERENCE

| Folder | One-Line Purpose | Add Files Here When... |
|---|---|---|
| `app/` | Pages and API routes | Creating a new URL/page |
| `app/admin/(protected)/` | Auth-gated admin pages | Adding a new admin section |
| `app/api/` | HTTP endpoints | External service needs to call your server |
| `components/ui/` | Shadcn primitives | Running `npx shadcn@latest add` |
| `components/layout/` | Header, Footer, CTAs | Changing site-wide chrome |
| `components/sections/` | Page content blocks | Adding a new page section |
| `components/[feature]/` | Feature UI | Building UI for a specific domain |
| `components/shared/` | Cross-feature UI | A component is used by 2+ features |
| `features/[feature]/` | Business logic | Adding a new server action or DB query |
| `lib/` | Third-party client setup | Integrating a new external service |
| `hooks/` | Client-side state | Extracting reusable hook logic from a component |
| `config/` | Static data | Adding a new constant that won't change at runtime |
| `types/` | TypeScript types | Defining shared types used across features |
| `prisma/` | Database structure | Changing the DB schema |
| `public/` | Static assets | Adding images, fonts, or static files |

## THREE RULES THAT PREVENT FUTURE MESS

**1. One source of truth per constant**
The city list lives in `config/cities.ts` only. Sitemap, service area pages, footer — all import from there. If you change a city name, you change it in one place.

**2. One Zod schema per form**
`features/leads/validations.ts` exports `LeadFormSchema`. The contact form uses it. The API route uses it. If the form changes, the validation changes in one place.

**3. Never put business logic in `lib/`**
`lib/prisma.ts` creates the Prisma client. It does NOT query the database. Database queries live in `features/[feature]/queries.ts`. This keeps infrastructure and logic separated.
